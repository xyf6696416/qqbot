# 意图路由 — 需求分析

## 1. 问题背景

桥接 v5 当前处理流程：

```
NapCat 消息 → 解析(文本/图片/合并转发)
           → prompt 拼接(system prompt + skill 指令 + 上下文)
           → OpenClaw API（每次都走完整 agent）
           → 从回复提取 MEDIA/EMBED 标签 → 执行发图等操作
```

**三个核心问题：**

### 问题 A：发图绕远路 🔴

用户说"发张图"时：
1. 桥接把整个系统 prompt（含大量发图指令）+ SKILL.md + TOOLS.md 传给 OC
2. Agent 读指令 → 理解意图 → 决定调用 `pick_fwd_image.py` → `send_image.py`
3. 回复走到桥接，桥接再解析 MEDIA 标签执行

**浪费**：每次发图多一次完整的 OC roundtrip（约 3-10 秒 + token 消耗），只为了得到一句"来啦~"和一个 MEDIA 标签。

### 问题 B：文生图无路由 🔴

用户说"生成一张猫的图片" → 目前没有"生图"这个能力入口
如果要加，不能跟"发图"（从本地选）和"识图"（看用户发的图）混在一起。

### 问题 C：识图硬触发 🔶

当前只要有 `img_urls`（用户发了图片附件）就走识图流水线。
但用户发图也有可能只是想**保存图片**（存图意图），而不是要 AI 识别。
目前靠 OC 第二次调用来判断，效率低下。

---

## 2. 目标架构

```
NapCat 消息 → 消息解析
           → 意图路由（新增层）
           → 按意图分流处理
           → 回复群聊
```

### 意图分类

| 意图 | 代号 | 说明 | 当前处理 | 目标处理 |
|------|------|------|----------|----------|
| **普通对话** | `chat` | 日常聊天、问问题、闲聊 | OC 完整处理 | 不变 |
| **发图** | `send_img` | "发张图"、"来张萝莉"、"发色图" | OC 处理 + 桥接解析 | **桥接直连 + 可选轻量 OC** |
| **识图** | `vision` | 用户发图问"这是什么" | 桥接特定检测 | **路由显式分类，覆盖更多 trigger** |
| **生图** *(未来)* | `gen_img` | "生成一张 xxx" | 无 | 桥接分流到图像生成 API |
| **存图** | `save_img` | "这张图存下来" | OC 告诉桥接执行 | 路由直接识别 + 执行复制 |

---

## 3. 各意图需求详解

### 3.1 普通对话 (`chat`)

**流程不变**，只是更干净：
```
router → chat → prompt拼接 → OC API → 回复
```

**增强建议**：
- 移除系统 prompt 中的大量发图/识图指令（技能文件不用全量拼接）
- 只注入当前意图需要的指令，减少 token 消耗
- 保留群 prompt + 上下文

### 3.2 发图 (`send_img`)

**需求分解**：

用户说发图时，可能有以下子意图：

| 用户说 | → 源文件夹 | → 数量 |
|--------|-----------|--------|
| "发张图" / "来张图" | 根目录（随机） | 1 |
| "发三张图" | 根目录 | 3 |
| "发张萝莉" | `--src 萝莉` | 1 |
| "发两张脚的色图" | `--src 脚` | 2 |

**目标处理流程**：

```
router detect "发图" intent
  ↓
解析关键词 → 确定 --src 和数量
  ↓
直接调用脚本（不进 OC）:
  1. python pick_fwd_image.py N --src <xxx>
  2. Copy-Item 到 tc/1/
  3. python send_image.py --group <gid> --url <图床url>
  ↓
[可选] 调 OC 生成一句回复语（轻量，不带技能指令）
  "来啦~" / "这张好看吗" / "给你" → 附在图后发出
```

**子问题**：关键词解析

需要维护一个关键词 → --src 映射表（目前 TOOLS.md 有，可移入 config.yaml）：

```yaml
intent_routing:
  send_img:
    keyword_map:
      萝莉: "萝莉"
      同人: "同人"
      色彩: "色彩图"
      脚: "脚"
      轮奸: "轮奸"
      大雷: "大雷"
    fallback_order: ["萝莉", "脚", "轮奸"]  # 根目录空时尝试顺序
    max_per_msg: 6                          # 单条消息最多6张
```

### 3.3 识图 (`vision`)

**当前触发条件**：消息中有 `img_urls`（用户发了图片附件）

**增强后的触发条件**：

```
1. 消息中含有图片附件 → vision
2. 用户引用/回复带图片的消息 + 文字含 "这是啥" "看看" "图里" 等
3. 消息文本含 "看图" "识图" "识别这个" 等
```

**目标处理流程**：

```
router detect "vision" intent
  ↓
用户带了图片 + 文字问题
  ↓
SiliconFlow Vision API → 原始描述
  ↓
[可选] OC 润色（带群 prompt，轻量）
  ↓
回复群聊
```

**特殊子场景**：
- 用户只发了图没说话 → 默认描述图片
- 用户发图 + "这是谁" → 带问题上下文发给 SiliconeFlow/OC
- 用户发图 + "发图"（冲突） → 发图优先（目前行为）

### 3.4 生图 (`gen_img`) — 未来预留

```
User: "生成一只猫" / "画一个赛博朋克城市"

router detect "gen_img" intent
  ↓
调用图像生成 API（DALL-E / Stable Diffusion / Midjourney）
  ↓
下载图片 → 复制到 tc/1/ → send_image.py
  ↓
[可选] 发图时带说明文字
```

**接口预留点**：
- 系统 prompt 中增加生图触发词
- config.yaml 增加 `gen_img` 配置块（API key、model、尺寸等）
- router 增加 `gen_img` 分支

### 3.5 存图 (`save_img`)

用户意图是把收到的转发图片存到桌面。

```
User: "存图" / "存到萝莉" / "这张存一下"
  ↓
router detect "save_img" intent
  ↓
确认转发图片已缓存 → Copy-Item 到桌面 \转发图片\
  ↓
回复"存好啦"
```

## 4. 路由实现方案

### 方案对比

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **A: 纯关键词** | 零成本、快、可靠 | 无法处理模糊意图 | **v1 首选** |
| **B: LLM 分类器** | 灵活、覆盖模糊输入 | 增加延迟+费用 | v2 可选 |
| **C: 混合（关键词 + LLM 兜底）** | 平衡 | 实现复杂 | 需要时再上 |

### 推荐：方案 A — 关键词路由

**设计原则**：
1. 基于关键词 + 简单正则匹配，`O(1)` 时间复杂度
2. 可配置（keyword_map 放 config.yaml）
3. 优先级明确：显式关键词 > 默认行为
4. 命中冲突时：发图 > 识图 > 普通对话

**匹配优先级（从高到低）**：

```
1. 命中 "生图" 关键词 → gen_img  (future)
2. 命中 "发图/来张/来点" 关键词 → send_img
3. 命中 "存图/保存" 关键词 → save_img
4. 消息含图片附件 → vision
5. 引用带图片的消息 + 含 "这是啥/看看" → vision
6. 其他 → chat
```

**关键词配置示例（config.yaml）**：

```yaml
intent_routing:
  enable: true
  
  # 关键词 → 意图映射
  keywords:
    # 发图（支持 "萝莉色图"、"脚的色图" 等带修饰的关键词）
    send_img:
      - ["发图", "发张", "来张", "发色图", "来点"]
      - ["发.*萝莉", "萝莉.*图", "来.*萝莉"]     # 正则
      - ["发.*脚", "脚.*图", "来.*脚"]
      - ["发.*大雷", "大雷.*图"]
      - ["发.*轮奸", "轮奸.*图"]
    
    # 识图
    vision:
      - ["这是啥", "这是什么", "看看", "图里", "看图"]
      - ["识别", "这个是什么", "帮我看看"]
    
    # 存图
    save_img:
      - ["存图", "保存", "存起来"]
      - ["存到", "保存到"]
    
    # 生图
    gen_img:
      - ["生成", "画一个", "画一张", "造一个"]
      - ["画.*猫", "生成.*图"]      # 精准匹配

  # 发图子文件夹路由
  send_img_subdir:
    萝莉: ["萝莉", "萝莉色图", "小萝莉"]
    脚: ["脚", "脚的色图", "玉足"]
    轮奸: ["轮奸", "轮奸图"]
    大雷: ["大雷", "大雷图"]
    同人: ["同人", "同人图"]
    色彩图: ["色彩", "色彩图"]
    
    # 无匹配时的 fallback
    fallback_order: ["萝莉", "脚", "轮奸"]
```

### 5.1 bridge.py 修改

**新增模块**：`IntentRouter` 类

```python
class IntentRouter:
    """
    意图路由：在消息处理进入 prompt 拼接前，
    根据消息内容和图片情况判断用户的意图。
    """
    
    def classify(self, text: str, has_image: bool) -> IntentResult:
        """返回意图分类结果，含子意图参数"""
        ...
```

**改造 `_process_queue_item`**：

```python
async def _process_queue_item(self, item):
    # 1. 解析消息
    text = item["text"]
    img_urls = item.get("img_urls", [])
    
    # ====== 新增：意图路由 ======
    intent = self.router.classify(text, has_image=bool(img_urls))
    
    # 2. 按意图分流
    if intent.type == "send_img":
        await self._handle_send_img(gid, uid, intent.params, at_bot)
        return
    elif intent.type == "vision":
        await self._handle_vision(gid, uid, text, img_urls, at_bot)
        return
    elif intent.type == "save_img":
        await self._handle_save_img(gid, uid, text, at_bot)
        return
    elif intent.type == "gen_img":
        await self._handle_gen_img(gid, uid, text, at_bot)
        return
    
    # 3. 普通对话（原有流程，简化系统 prompt）
    ...
```

**新增 handler 示例**：

```python
async def _handle_send_img(self, gid, uid, params, at_bot):
    """发图意图：桥接直连，不进 OC"""
    src = params.get("src")
    count = params.get("count", 1)
    
    # 选图 → 复制 → 发送
    paths = self._pick_images(src, count)
    if not paths:
        await self._send_text(gid, "没图啦，库存空了 😅")
        return
    
    self._copy_to_image_bed(paths)
    await self._send_images(gid, paths)
    
    # [可选] 轻量 OC 回复
    # reply = await self._call_oc_light(gid, "用户让我发图，我发了N张，简单回应一句")
    # await self._send_text(gid, reply)
```

### 5.2 config.yaml 新增

```yaml
intent_routing:
  enable: true
  keywords: { ... }
  send_img_subdir: { ... }
  send_img:
    max_per_msg: 6
```

### 5.3 SKILL.md 简化为最小指令

发图/识图指令不再全量塞入 system prompt，只保留最核心的安全约束。
具体路由和操作逻辑由 bridge 层直接控制。

## 6. 非功能需求

### 6.1 性能

- 关键词匹配需在 **1ms 内完成**
- 涉及子进程调用（pick_fwd_image）需缓存/预热
- 发图流程目标：**从收到消息到发出图片 < 2 秒**（当前走 OC 约 5-15 秒）

### 6.2 可观测性

- 每条消息记录意图分类结果 → 日志 `INTENT: send_img src=萝莉 count=2`
- 统计各意图频次 → 了解用户使用模式
- 发图失败报警

### 6.3 容错

- 关键词未命中时回退到 `chat` 意图
- 脚本执行失败时回退到聊天回复告知用户
- 意图已废弃时输出日志但不报错

## 8. 风险与边界

| 风险 | 影响 | 缓解 |
|------|------|------|
| 关键词误命中（如"看看"在普通对话中） | 错误进入识图分支 | 仅当消息同时有图片/引用图才触发 vision |
| 用户说"发消息给他们看看" | 误判为发图 | 提高发图关键词特异性 |
| 发图脚本失败 | 用户收到无回复 | 加 try-catch，失败走 OC 兜底 |
| 关键词列表维护成本 | 漏配新关键词 | 支持正则 + 定期 review 日志 |

## 9. 总结

```
当前（v5）                      目标
───────                        ────
消息                           消息
  ↓                              ↓
prompt 拼接（含所有技能）        意图路由（1ms 内分类）
  ↓                              ↓
OC API（完整 agent 推理）        发图 → 直连脚本（<2s）
  ↓                              识图 → vision + 轻量 OC
执行 MEDIA/EMBED 标签             存图 → 直连复制
                                 生图 → 调用 API（未来）
                                 对话 → 精简 prompt + OC
                               ↓
                              OC 只处理需要它的事
                              token 消耗减少 30-50%
                              发图响应时间 15s → 2s
```
